Alert - The Debugging Dark Ages


function onAlertClick() {
  alert('alert text was clicked');
}
  • Available everywhere.
  • But, it's visible to the user and interrupts execution

console.log - Alert the Developers!


var clickCount = 0;
function onLogClick() {
  clickCount++;
  console.log('log text was clicked ' + clickCount);
}
  • A better alert, is logged to developer console and doesn't interrupt execution.
  • But, beware IE8/9. Calling console.log will throw an exception unless developer tools have been opened.
  • To fix IE8/9, include https://github.com/paulmillr/console-polyfill

JS Debuggers - Now that's Power

  • Breakpoints
  • Step into and step out
  • Call stack. For IE8, you'll want to use named function expressions or function declarations for a cleaner call stack
  • Watch window, both variables and expressions.
  • Mouse over variable. Works in IE9+, Chrome and Firefox
  • Mouse over expression. Works in IE9+ and Chrome

JS Debuggers - Break On Demand?

  • Conditional breakpoints - debugger should only stop at breakpoint when JS expression is true.
  • debugger; statement - when this statement is run, browser will stop and enter debugger.
  • Both are very useful for allowing you to skip past the boring stuff on your given breakpoint.

JS Debuggers - Logpoint/Tracepoint

  • Remember conditional breakpoints? Remember console.log?
  • Use console.log as the conditonal expression for a breakpoint. It returns falsey so debugger never stops, but it always runs your console.log statement.
  • Really useful to add console.log to third-party code, like jquery, Angular, Knockout, etc.
  • Also really useful for debugging recursive code or complicated event handling like mouse in.

JS Debuggers - Exceptions

  • Stop when an uncaught exception escapes.
  • Stop at the exact point an exception is thrown. Works in IE11, Chrome, Firefox. IE8-10, it will stop but you might be missing frames in the call stack.. Really useful for catching: 
    
try {
// some code here...
}
catch () {}

Console - the swiss army knife

  • The "Console" tab of the developer tools.
  • Remember, the default for a JS variable is a global variable.
  • And the JS vm is initialized per-page.
  • Everything you type in the console runs in the JS vm for the page, which means it has access to everything in the page. IE11, Chrome work with sub-frames on a page.
  • So, the possibilities are limited by your imagination and closures.

Console - More details

  • Retains history of previously entered JS, use up and down arrow.
  • Supports copy/paste and multiple lines. IE has single-line and multi-line mode, Chrome and Firefox you must paste multiple line code into console.
  • Intellisense completion using 'tab'. Works in IE11, Chrome and Firefox
  • Expand values of object and arrays directly in console. Works in IE11, Chrome and Firefox
  • Interacts with "Elements" tab of developer tools using $0, $1 variables.Works in IE11, Chrome and Firefox
